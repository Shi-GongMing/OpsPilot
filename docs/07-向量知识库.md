# 07 — 向量知识库全链路：文档到向量的完整旅程

> **面试重点**：Embedding 全链路设计、文档分割策略、元数据传递
>
> **涉及文件**：`api/file.py` → `services/vector_index_service.py` → `document_splitter_service.py` → `vector_embedding_service.py` → `vector_store_manager.py`

---

## 0. 全链路数据流

```
POST /api/upload (file)
  → 保存到 uploads/
  → VectorIndexService.index_single_file()
      → 读文件内容
      → 删除该文件的旧索引（覆盖更新）
      → DocumentSplitterService.split_markdown()
          → 按 #/## 标题分割（MarkdownHeaderTextSplitter）
          → 按大小二次分割（RecursiveCharacterTextSplitter, chunk=800）
          → 合并过小片段（<300 字符）
          → 添加元数据（_source, _extension, _file_name, _indexed_at）
      → VectorStoreManager.add_documents()
          → DashScopeEmbeddings.embed_documents() → OpenAI API
          → Milvus.insert() → L2 索引
```

## 1. 文档分割：两阶段 + 合并优化

### 为什么分两阶段

Markdown 有标题结构——如果单纯按字符数切，可能把"## 告警名称"和其下的内容切成两块。**第一阶段的标题分割**保留了文档的语义结构，每个分片有明确的 h1/h2 标题。**第二阶段的递归字符分割**确保单个分片不会太大。

### 小片段合并

分割后可能出现极小的片段（如只有标题没有内容的片段）。合并逻辑：小于 300 字符的片段，如果和上一个片段合并后不超过 `chunk_size × 2`（1600 字符），则合并。

**重要细节**：合并时用 `deepcopy` 创建新对象——如果直接修改原始 Document 对象，会导致数据被意外篡改。

### 面试一句话

> "分割策略是两阶段——先按 Markdown 标题保留语义边界，再用递归分割控制粒度。太小的小片段（<300 字符）会合并到相邻片段，避免碎片化。合并时做了深拷贝防止修改原始对象。"

### [自研概念] MarkdownHeaderTextSplitter / RecursiveCharacterTextSplitter

## 2. Embedding 服务：LangChain 标准接口

实现了 LangChain 的 `Embeddings` 抽象类的两个接口：

- `embed_documents(texts)` → 批量嵌入，用于文档入库
- `embed_query(text)` → 单个嵌入，用于用户查询

底层调用阿里云 DashScope 的 text-embedding-v4 模型（1024 维），使用 OpenAI 兼容接口。

**延迟初始化设计**：如果 API Key 未配置，构造函数不抛异常，只记录 warning 并将 client 置为 None。首次调用 `embed_documents/embed_query` 时，通过 `_ensure_client()` 检查 client 是否就绪——未就绪则抛出明确的 RuntimeError。这样应用在 API Key 缺失时仍然能够启动，只是 Embedding 功能不可用。

## 3. Milvus 存储管理

### Collection 设计

- Collection 名称：`biz`（所有业务数据统一存储）
- 索引类型：L2 距离
- 关键参数：`auto_id=False`（自定义 UUID 主键）、`text_field="content"`、`vector_field="vector"`

### 元数据设计

每个 chunk 携带的元数据：
| 字段 | 作用 |
|------|------|
| `_source` | 源文件路径（用于覆盖更新） |
| `_file_name` | 文件名（检索结果展示） |
| `h1`, `h2` | 标题层级（检索结果结构化展示） |
| `_indexed_at` | 入库时间（时间衰减检索） |
| `_source_type` | `manual_upload` / `auto_archive` / `context_summary` |

### 覆盖更新

同文件重新上传时，先用 `metadata["_source"] == "path"` 表达式删旧数据，再插入新数据。保证不会出现重复。
