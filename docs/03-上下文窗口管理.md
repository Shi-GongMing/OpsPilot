# 03 — 上下文窗口管理：长对话如何不失控

> **面试重点**：四层上下文管理架构、滑动窗口/摘要/向量检索的组合策略、溢出分级降级
>
> **涉及文件**：`services/context_manager.py` → `services/rag_agent_service.py`（接入点）

---

## 0. 行业标准框架

2025-2026 年面试八股文对这一块形成了标准回答框架——四层上下文管理：

```
第 1 层：窗口分配 — 给各部分设 token 配额，监控实际消耗
第 2 层：记忆分层 — 工作记忆(窗口内) / 短期记忆(摘要+向量检索) / 长期记忆(持久化)
第 3 层：压缩策略 — 滑动窗口 + 摘要压缩 + 向量检索归档 + 工具输出脱水
第 4 层：上下文编排 — 推理链压缩 + 子 Agent 隔离 + 溢出降级兜底
```

**核心矛盾**：既要保留历史信息（不让 Agent 失忆），又要控制上下文大小（不让延迟和成本失控）。

---

## 1. 第 1~2 层：窗口分配 + 记忆分层

### 窗口分配的核心观点：不看数据调上下文等于盲调

给每类内容设 token 预算（示例）：

| 内容类型 | 配额 | 作用 |
|----------|------|------|
| System Prompt | ≤2K | 角色定义 |
| 工具定义 | ≤5K | Function Calling schema |
| 对话历史 | ≤32K | 多轮上下文 |
| RAG 文档 | ≤40K | 知识库召回 |
| 工具返回 | ≤32K | MCP/本地工具结果 |

### 记忆分层的三层模型

| 层级 | 存储 | 生命周期 | 检索方式 |
|------|------|---------|---------|
| 工作记忆 | LLM 上下文窗口内 | 当前对话 | 已在 prompt 里 |
| 短期记忆 | 向量库 + 摘要 | 同一 session | 语义检索 |
| 长期记忆 | 关系库 / 向量库 | 跨 session | 用户 ID + 语义 |

> 面试金句："工作记忆放窗口即用即抛，短期记忆用摘要+向量检索，长期记忆持久化按需加载。"

---

## 2. 第 3 层：四种压缩策略详解

### ② 摘要压缩 — 最常用，但要区分"压缩"和"摘要"

| 操作 | 可逆性 | 做法 |
|------|--------|------|
| 压缩 (Compaction) | 可逆 | 原文外部化到文件/数据库，窗口里只放索引指针，需要时读回原文 |
| 摘要 (Summarization) | 不可逆 | LLM 精炼为一段文字，原始措辞永久丢失 |

面试加分点：说一句"我们优先用可逆压缩，只在必须时做摘要"。

**摘要最容易丢失的五类信息**：
1. 精确数值（阈值、端口、版本号）
2. 硬约束（"绝对不碰 test 文件"）
3. 决策推理过程（为什么选 A 不选 B）
4. 跨任务依赖（第 12 轮的修改影响第 47 轮）
5. 隐式偏好（编码风格、回答详细程度）

对策：这五类在每轮对话后提取到结构化存储，**在压缩发生前**保存。

### ③ 向量检索归档 — 最关键的一项

**入库的是原始消息本身，不是摘要。**

逻辑：历史消息 embedding → 存向量库。用户问新问题时，不光看当前窗口的消息，还向向量库发语义搜索，把跟当前问题最相似的历史消息**原样**召回，拼进 prompt。

三种存放方式互补：

| 位置 | 形式 | 优点 | 缺点 |
|------|------|------|------|
| 窗口内 | 最近 N 条原文 | 精确即时可用 | 数量有限 |
| 向量库 | 历史原文 embedding | 按需语义召回、不丢原文 | 需 embedding 基础设施 |
| 窗口内 | LLM 摘要 | 空间极小 | 损失精确信息 |

## 3. 第 4 层：溢出降级兜底（面试必问）

**"如果压缩了还是溢出怎么办？"** 标准答案是分级降级：

| 优先级 | 策略 | 操作 |
|--------|------|------|
| P1 | 裁剪工具返回 | 砍掉非关键字段 |
| P2 | 压缩对话历史 | 旧消息替换为摘要，保留最近 3 轮原文 |
| P3 | 减少 RAG 文档 | Top-5 降到 Top-2 |
| P4 | 简化系统指令 | 去掉非核心 Prompt |
| P5 | 终止当前任务 | 提示开新会话，**但保留摘要传递** |

预警机制（体现工程意识）：
> "黄色预警（剩余 20% 空间）启动摘要预计算；橙色（剩余 15%）执行结构化存储——把精确值和约束提取到外部；红色（剩余 10%）强制压缩并通知用户。"

## 4. 我们项目中的实现

基于上述标准框架，OpsPilot 中的落地：

**实现机制**：继承 LangGraph 的 `AgentMiddleware`，重写 `abefore_model` 钩子。这是每次 LLM 调用前触发的标准拦截点。未超过 16 条消息时返回 `None` 不干预，超过后执行压缩。

**已实现的能力**：
- 滑动窗口：最近 10 条完整保留
- 摘要压缩：溢出消息中最近 8 条经 LLM 压缩为 200 字结构化摘要，注入到消息列表头部
- 向量归档：摘要写入 Milvus（`context_summary` 标签标记）
- 摘要去重：多轮压缩后只保留最新一条摘要消息

**可进一步对齐行业标准的点**（面试时作为"后续优化方向"讨论）：
- 按 token 而非消息数计数
- 可逆压缩（原文外部化后再做摘要）
- 完整的 P1-P5 降级链条

## 面试话术

> "我不只用滑动窗口。标准做法是四层——窗口分配给各部分设配额，记忆分层区分工作/短期/长期，压缩策略组合滑动窗口+摘要+向量检索归档，编排层做推理链压缩和溢出降级。
>
> 核心思路是'冷热分离'——当前需要的放窗口，可能需要的放摘要，未来可能用到的存向量库语义召回。另一个关键：压缩不等于摘要，可逆压缩保留原文指针，摘要不可逆。摘要最容易丢五类东西——精确数值、硬约束、决策原因、跨任务依赖、隐式偏好——这些要在压缩前提取到结构化存储。
>
> 溢出兜底有分级降级：先裁冗余字段→再压对话→再减文档→最后降级提示词，实在不行终止任务但保留摘要让用户无缝切换。实现上我们继承 LangGraph 的 AgentMiddleware，重写了 abefore_model 钩子做压缩和归档。"
