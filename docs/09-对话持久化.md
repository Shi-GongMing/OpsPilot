# 09 — 对话持久化：关机重启后对话还在

> **面试重点**：LangGraph checkpoint 机制、AsyncSqliteSaver 的选择与限制、主线程 async 限制的处理
>
> **涉及文件**：`core/checkpoint_manager.py`、`services/rag_agent_service.py`、`services/aiops_service.py`

---

## 1. 为什么要持久化

Agent 对话是**有状态**的——第 3 轮的"我服务器在北京"要能被第 10 轮的 Agent 引用。如果对话历史只存在内存中（`MemorySaver`），服务重启后全部丢失。

解决方案：把对话状态持久化到 SQLite（`AsyncSqliteSaver`），服务重启后通过 session_id 恢复完整历史。

---

## 2. LangGraph Checkpoint 机制

```
Agent.ainvoke(question, config={"configurable": {"thread_id": "xxx"}})
    │
    ├── 1. checkpointer.aget(config) → 有历史? → 恢复到 State
    ├── 2. LLM 推理 → 工具调用 → ...
    ├── 3. checkpointer.aput(config, checkpoint) → 写入新状态
    └── 4. 返回最终结果
```

**核心概念**：`thread_id` 是会话标识。同一个 thread_id 的所有 `ainvoke` 调用共享同一个 checkpoint 链——每次调用追加消息，LangGraph 自动处理状态的读取和写入。

**Checkpoint 存储结构**：

| 表 | 内容 |
|----|------|
| `checkpoints` | `thread_id` + `checkpoint_id` + `msgpack` 二进制数据 |
| `writes` | 每次状态变更的增量记录 |

序列化格式是 msgpack（二进制，比 JSON 更紧凑高效）。

---

## 3. 为什么用 AsyncSqliteSaver 而不是 SqliteSaver

LangGraph 提供了两个 SQLite 持久化方案：

| 方案 | 线程模型 | 限制 |
|------|---------|------|
| `SqliteSaver` | 同步，适合独立线程 | 主线程调用会阻塞事件循环 |
| `AsyncSqliteSaver` | 异步，基于 aiosqlite | **主线程必须用 async 方法** |

`AsyncSqliteSaver` 在主线程中强制使用 `aget_tuple`/`aput`/`adelete_thread` 等 async 方法——如果在主线程调用同步的 `get_tuple`，会抛 `InvalidStateError`。

这是因为 aiosqlite 在内部使用独立工作线程处理 SQLite 操作，主线程的 async 方法通过事件循环与工作线程通信。如果我们用 `asyncio.to_thread` 把调用外包出去可以绕过这个限制，但官方推荐直接用 async 接口。

## 4. 延迟初始化

```python
# 构造时不连数据库——__init__ 是同步方法
self.checkpointer = None

# 首次对话请求时异步初始化
async def _ensure_checkpointer(self):
    if self.checkpointer is None:
        self.checkpointer = await get_checkpointer()
```

**为什么不在 `__init__` 中初始化**：`AsyncSqliteSaver.setup()` 和 `aiosqlite.connect()` 都是 async 操作，不能在同步 `__init__` 中调用。这是 async-first 架构的标准模式。

---

## 5. 全局单例

RAG Agent 和 AIOps Service 共享同一个 AsyncSqliteSaver 实例——两者把 `session_id`（对话）和 `thread_id`（诊断 session_id）分开，互不干扰。共享实例的好处是 SQLite 只有一条数据库连接，避免锁竞争。

---

## 6. 会话管理 API

| 接口 | 功能 | 实现 |
|------|------|------|
| `GET /chat/session/{id}` | 查询会话历史 | `aget_tuple(config)` → 反序列化 messages |
| `GET /chat/sessions` | 列出所有会话 | SQL 查 checkpoints 表 → 去重 thread_id → 逐个读消息数 |
| `POST /chat/clear` | 清空指定会话 | `adelete_thread(thread_id)` |

> "对话持久化用了 LangGraph 的 AsyncSqliteSaver。核心原理是每次 Agent 推理完成后，框架自动通过 checkpointer 把消息历史序列化为 msgpack 写入 SQLite——thread_id 作为会话标识，同一 thread 的消息自动累加。我们遇到了一个工程问题——AsyncSqliteSaver 在主线程中只能用 async 方法，所以初始化必须延迟到首次异步请求，不能在 `__init__` 中完成。全局单例让 RAG Agent 和 AIOps 服务共享同一个连接。"
