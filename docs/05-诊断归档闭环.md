# 05 — 诊断归档闭环：Agent 如何"自我进化"

> **面试重点**：自演进闭环的设计理念、fire-and-forget 归档策略、覆盖更新
>
> **涉及文件**：`services/diagnosis_archiver.py`、`services/aiops_service.py`（归档触发点）

---

## 1. 设计理念

**核心创新**：把知识库从"人工手动上传文档"升级为"Agent 自主沉淀经验"。

```
诊断 → 生成报告 → 结构化拼接 → 分块向量化 → 写入 Milvus
    → 下次 Planner 检索时命中归档案例 → 诊断越来越准
```

每次诊断都在增强系统的知识储备。这与 RAG 的上传文档机制形成互补——手动上传是"教科书知识"，自动归档是"临床经验"。

---

## 2. 归档触发时机

诊断流程在 `aget_state()` 获取最终状态后触发归档：

```python
# 线程池异步执行，不阻塞 SSE 推送给前端
asyncio.create_task(
    asyncio.to_thread(archive_diagnosis, session_id, user_input, past_steps, final_response)
)
```

**为什么用 `to_thread` 而非直接 async 调用**：归档链路中的 `vector_store_manager.add_documents()` 是同步 Milvus API。放线程池执行，不阻塞主事件循环的 SSE 推送。

**为什么是 fire-and-forget**：归档失败不应影响诊断报告的推送——用户不应该因为"归档写库失败"而看不到诊断结果。如果归档出错，日志记录后静默丢弃。

---

## 3. 归档文档的结构化

不是把原始报告直接入库——而是拼接为标准故障文档，方便检索命中和人工查阅：

```markdown
# 自动归档故障案例
- 归档时间: 2026-06-14T19:35:09
- 会话 ID: archive-verify-001
- 来源: AIOps 自动诊断归档

## 诊断任务
(原始诊断任务描述)

## 排查过程
### 步骤 1: ...
(执行结果，过长截断 2000 字符)

## 诊断结论
(完整的 final_response)
```

**为什么不让 LLM 重新精炼**：当前信息已足够结构化，模板拼接零额外成本。如果让 LLM 再总结一次，增加一次 API 调用（时间+费用）。后续版本可以加 LLM 精炼作为可选增强。

---

## 4. 覆盖更新策略

同一 session 多次诊断会覆盖旧归档：

```python
deleted = vector_store_manager.delete_by_source(archive_file_name)
if deleted > 0:
    logger.info(f"已删除旧归档 ({deleted} 条)")
```

删旧写新。保证同一次诊断不会在库中留下多份过时记录。

---

## 5. 元数据标记设计

归档文档的元数据包含：
- `_source_type: "auto_archive"` —— 与手动上传（无此标记）区分
- `_diagnosis_time: ISO 时间戳` —— 供时间衰减检索使用
- `_source: "auto_archive/{session_id}.md"` —— 虚拟文件路径，支持覆盖更新

`_source_type` 的设计让后续可以按类型检索——"只看自动归档的案例"或"只看手动上传的文档"。

---

## 6. 短路保护

```python
if not final_response or len(final_response.strip()) < 50:
    return  # 报告太短，不归档
```

诊断异常短路时产生的空报告或极短报告，不污染知识库。

---

## 面试话术

> "我们设计了一个自演进的知识闭环——每次诊断完，自动把故障现象、排查步骤、根因结论和解决方案拼接为标准文档，分块向量化后写入 Milvus。下次 Planner 制定计划时会检索历史案例作为参考——系统排查的故障越多，经验越厚。
>
> 几个工程细节：归档用 asyncio.to_thread 丢到线程池，不阻塞诊断报告的 SSE 推送。失败静默记录不抛异常——用户不应该因为归档失败而看不到诊断结果。同一 session 重复诊断会覆盖旧记录。报告不足 50 字符不归档，避免噪声入库。"
