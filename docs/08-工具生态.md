# 08 — 工具生态：Agent 的"手和脚"

> **面试重点**：MCP 协议的理解、工具降级策略、Function Calling 的底层机制
>
> **涉及文件**：`agent/mcp_client.py`、`tools/`、`mcp_servers/`

---

## 1. 工具全景

| 类别 | 数量 | 工具 | 来源 |
|------|:---:|------|------|
| 本地工具 | 3 | 知识检索、Prometheus 告警、时间查询 | `app/tools/`，`@tool` 装饰器 |
| MCP 远程工具 | 7 (CLS 5 + Monitor 2) | 时间戳、日志搜索、topic 查找、CPU/内存指标等 | `mcp_servers/`，通过 MCP 协议暴露 |

总共 10 个工具统一注册给 Agent，Agent 通过 Function Calling 自主选择调用哪个。

---

## 2. MCP 协议是什么

MCP（Model Context Protocol）是 Anthropic 提出的**工具标准化协议**。核心思想：工具提供方（MCP Server）和工具使用方（MCP Client）之间通过标准协议通信，与具体 LLM 或框架解耦。

### 为什么用 MCP

| 传统方式 | MCP 方式 |
|---------|---------|
| 工具硬编码在 Agent 代码中 | 工具独立部署为服务，通过标准协议接入 |
| 新增工具需要改 Agent 代码并重新部署 | 新增 MCP Server 即可，Agent 自动发现新工具 |
| 工具用各种协议和格式 | 统一的 JSON-RPC over HTTP/stdio |

### [自研概念] MCP 协议 / JSON-RPC

## 3. MCP 客户端的设计细节

### 单例模式 + 竞态保护

```python
async with _mcp_lock:
    if _mcp_client is None:
        _mcp_client = _create_mcp_client(...)
```

全局只一个 MCP 客户端实例——10 个工具通过同一个连接管理生命周期。`asyncio.Lock` 防止并发首次请求时创建多个实例。

### 重试拦截器

```
工具调用失败 → 等待 delay → 重试
  还失败 → delay × 2 → 再重试
  还失败 → delay × 4 → 最后重试
  全部失败 → 返回错误结果（不抛异常）
```

指数退避（1s → 2s → 4s），最多 3 次。全失败后返回带 `isError=True` 的结果——Agent 可以基于错误信息换工具或调整策略，不会崩溃。

### 降级策略

`load_mcp_tools_safe()` 加载 MCP 工具时如果失败，返回空列表——Agent 至少能用 3 个本地工具继续运行。

## 4. Function Calling 的底层机制

面试时被问"Function Calling 是怎么工作的"，标准答案分四步：

1. **工具定义注入 prompt**：`bind_tools(all_tools)` 将每个工具的 name、description、parameters schema 作为 system message 的一部分发给 LLM。这是 OpenAI 定义的 Function Calling 协议——LLM 读到工具描述后，如果判断需要调用，会在回复中带上 `tool_calls` 字段。

2. **LLM 决策**：LLM 根据用户问题和工具描述，自主判断"是否需要调工具"以及"调哪个工具"。模型要么返回纯文本（不需要工具），要么返回 `tool_calls` 列表（需要工具）。

3. **框架执行工具**：LangGraph 的 `ToolNode` 收到 `tool_calls` 后，按 name 匹配工具函数，解析 args 参数，执行工具，将返回结果包装为 `ToolMessage` 追加到消息历史。

4. **LLM 基于结果继续推理**：`ToolMessage` 被送回 LLM，LLM 看到工具返回的数据后继续推理——可能再调工具，也可能给出最终答案。

---

## 5. 工具定义的最佳实践

每个工具的 `name` 和 `description` 直接影响 LLM 的调用决策。name 要简洁明确（"query_cpu_metrics"），description 要说清楚"什么场景下用、返回什么数据"。

工具内部的异常处理全部 try/except 并返回错误字符串——不让工具抛出的异常打断 Agent 的推理流程。

---

## 面试话术

> "工具分两类——3 个本地工具直接注册，7 个远程工具通过 MCP 协议接入。MCP 的好处是工具和 Agent 解耦——新增一个工具只需部署一个新的 MCP Server。客户端做了单例模式加 asyncio.Lock 防止并发创建，指数退避重试保证偶发网络抖动不会导致诊断中断。MCP 加载失败时降级为纯本地工具——Agent 能力可有损但不会整体不可用。"
